DeOpenRouter

基于 Solidity、Next.js、Hono 与 FastAPI 的透明、低信任假设 AI API 市场原型：链上负责信任协调、定价、支付与审计锚定；模型推理保留在链下。

本页约 0:45 · Opening / Rubric

Story · 叙事动机

中心化 AI 网关的信任负担

传统网关往往把定价、路由、计费与争议处理集中交给单一运营方：方便，但用户难以独立验证规则是否被悄悄改写、扣费是否与公开策略一致。

  • DeOpenRouter 的叙事起点：哪些环节适合迁到 EVM，让用户可观察、可对账、难单方面篡改？
  • 不假装「推理全部上链」，在现实可用性与结算透明之间做可辩护的取舍。

本页约 1:35 · Story

Story · 链上迁移的信任边界

项目目标（README 归纳）

  • 提供方身份承诺与元数据在链上公开登记
  • 调价非即时生效，带有可观察延迟
  • 每次付费调用留下可审计链上记录
  • 质押可通过带挑战期的链上罚没提案削减；罚没款进入可配置 slashTreasury
  • 外部审计结果以报告哈希形式锚定上链

推理仍在链下：保留可用性，同时让结算与审计过程更透明。

本页约 0:50 · Implementation

Story + Implementation · 主流程

工作流程（五步）

  1. 提供方链上注册：价格、质押、元数据 URI、身份/能力哈希、endpointCommitment
  2. Web 展示市场；用户可通过 relay API 先模拟请求
  3. 前端对请求与响应做哈希，随付款发起链上 invoke(...)
  4. 合约向提供方转费、保存调用记录、发出 CallRecorded 事件
  5. 若启用审计：链下探测 relay → 规范 JSON（canonical）→ 可选链下发布 → recordAudit / recordAuditWithUri 锚定 keccak256(canonicalJson)

本页约 1:15 · Story + Implementation

Implementation · Solidity

核心合约：DeOpenRouterMarketplace.sol

主合约位于 contracts/src/。下列函数名与 README 一致，便于对照代码与评分中的「智能合约实现」。

  • register — 带质押注册新提供方
  • updateProviderMetadata — 更新 metadataURI、metadataHash、identityHash
  • announcePriceChange / getEffectivePrice — 声明未来价格；只读当前有效价
  • getProviderCore — 紧凑读取 owner、价格、质押与罚没元数据
  • invoke — 收款、记哈希、发 CallRecorded
  • deactivate + withdrawStake — 停用与锁定期后提取质押

README 警告：经济安全与信任假设

经济安全仍取决于多签/金库配置。链上已实现罚没延迟 + 挑战与审计员白名单多签，但尚无完整仲裁市场、托管结算或推理密码学证明。

本页约 0:35 · Story

Story · MVP 边界

当前不承诺的能力（README）

  • 链上仅存 endpointId 承诺，不公开真实 endpoint URL
  • 调用记录仅存请求/响应哈希，无明文全文
  • 单次即时结算；无托管、流式或异步结算
  • 罚没提案由 slashOperator 发起，执行有挑战期；多审计员法定人数在链下聚合
  • 无去中心化发现、信誉系统、自动化仲裁市场或推理执行证明网络
